## 1. 개요

실제 운영 환경을 가정한 작업을 설계하고 구현하는 능력을 확인하기 위한 목적입니다.
단순 기능 구현이 아니라, 불확실성이 존재하는 환경에서의 설계 사고가 필요합니다.

---

## 2. 시나리오

클라이언트(프론트엔드)는 **현재 서버**에 이미지 처리 요청을 보냅니다.
지원자의 서버는 **외부 이미지 처리 서비스(Mock Worker, 이하 \"Mock Worker\")** 에 실제 작업을 위임합니다.  
이 서비스는 대용량 이미지를 대상으로 하는 AI 추론(딥러닝 모델)을 수행하며, GPU 리소스를 많이 소모하는 무거운 작업입니다.  
따라서 응답 시간은 수 초에서 수십 초까지 크게 변동될 수 있으며, 순간 트래픽이 몰릴 경우 안정적으로 응답을 받기 어렵습니다.
이미지 처리는 시간이 오래 걸리는 작업이며, 요청 시점에 즉시 완료되지 않습니다.  

클라이언트는 작업의 진행 상황과 결과, 그리고 작업의 목록을 확인할 수 있어야 합니다.
---

## 3. 요구사항

### 3.1 기능 요구사항

1. 이미지 처리 요청을 수신하는 API를 구현해야 합니다.
   - 클라이언트가 작업을 식별하고 추적할 수 있는 방법을 제공해야 합니다.

2. 클라이언트가 작업의 진행 상태를 확인할 수 있어야 합니다.
   - 작업의 상태 모델은 스스로 정의해야 합니다.

3. 작업 완료 시 결과를 조회할 수 있어야 합니다.
   - 실패 상황에 대한 표현 방식 또한 정의해야 합니다.

4. Mock Worker와 연동하여 작업을 수행해야 합니다.

1. API Key 발급

엔드포인트:
POST /mock/auth/issue-key

요청 바디 예시:
{
  "candidateName": "string",
  "email": "string"
}

응답 예시:
{
  "apiKey": "mock_xxx"
}
발급받은 API Key는 이후 요청 시 헤더에 포함해야 합니다.

2. 이미지 처리 요청

엔드포인트:
POST /mock/process

요청 헤더:
X-API-KEY: {발급받은 apiKey}

이미지 데이터의 구체적인 표현 방식은 지원자가 자유롭게 설계할 수 있습니다.

3. Mock Worker 특성

Mock Worker의 내부 구현은 수정할 수 없습니다.

Mock Worker의 API 스펙(엔드포인트, 요청/응답 필드)은 변경되지 않습니다.
다만 실제 외부 SaaS와 마찬가지로 응답 시간과 안정성 등 내부 동작 특성은  
변경될 수 있으며, 사전에 예측하기 어렵다고 가정하고 시스템을 설계해야 합니다.

### 3.3 기술 조건

- Java 또는 Kotlin + Spring Boot 기반으로 구현해야 합니다.
- 테스트 코드를 포함해야 합니다.
- 애플리케이션은 컨테이너 환경에서 실행 가능해야 합니다.

## 4. 추가 요구사항

### 4.1 중복 요청 처리

- 동일한 요청이 여러 번 전달될 수 있는 상황을 가정해야 합니다.
- 이에 대한 처리 전략을 설계하고 설명해야 합니다.

### 4.2 상태 전이

- 전체 상태 전이 흐름을 정의해야 합니다.
- 허용되지 않는 상태 전이 또한 명확히 해야 합니다.

### 4.3 처리 보장 모델

- 본 시스템이 어떤 처리 보장 모델에 해당하는지 명시해야 합니다.
- 그렇게 판단한 근거를 설명해야 합니다.

### 4.4 서버 재시작 시 동작

- 서버가 재시작되는 경우 진행 중 작업이 어떻게 처리되는지 설명해야 합니다.
- 데이터 정합성이 깨질 수 있는 지점을 기술해야 합니다.
